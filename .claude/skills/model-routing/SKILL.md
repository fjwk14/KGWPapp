# モデル・サブエージェント運用

共有ルールは `docs/AI_WORKFLOW.md` を優先する。

## 大原則
- セッションのモデルはユーザーが `/model` で切り替える(アシスト側からは変更不可)。
- 同一セッション内でモデルを切り替えても文脈は保持される。切替コストはゼロ。
- サブエージェントは別モデルを指定できる(Agent の `model:` )が、**コールドスタートで文脈を読み直す**。小さな作業ほど割高。**大きく自己完結した塊**のときだけ使う。

## ルーティング表
| 種類 | 推奨モデル | 例 |
|---|---|---|
| 要所(設計・判断) | **fable** / `architect` | データモデル/RLS設計、曖昧なバグ調査、UX判断、未知領域、最終レビュー |
| 通常実装 | **sonnet** / `routine-implementer` | 方針確定後の画面・Server Action・テスト実装 |
| 探索・反復 | **haiku** / `Explore` / `test-runner` | 検索、依存関係確認、テスト/E2E反復、ログ要約 |

## 実践手順(ユーザー向け)
1. 新機能や難所は親セッションを `/model fable` にするか、読み取り専用の `architect` に調査させる。
2. 方針が決まった限定実装は `routine-implementer` へ委譲するか `/model sonnet` で実装する。
3. 探索は `Explore`、テスト反復は `test-runner` に委譲する。
4. 最終判断と統合は親セッションで行い、重要変更は `reviewer` に読み取り専用レビューさせる。

## 実践手順(アシスト向け: サブエージェント委譲)
次の条件を**すべて**満たすときだけ、定義済みサブエージェントに委譲する:
- タスクが自己完結していて、成功条件が明確(例: 「この3ファイルを新UIに合わせ、全E2Eをグリーンにして報告」)。
- 反復や大きな出力(テストログ等)が予想される。
- メインの会話文脈をほとんど必要としない(必要な情報はプロンプトに書ける)。

委譲プロンプトに必ず含める: 対象ファイル、成功条件、禁止事項、ローカルスタックの起動コマンド、検証コマンド、期待する報告形式、コミット/PRの要否。

## やってはいけない
- 小さな修正のためにサブエージェントを乱発する(コールドスタート分だけ損)。
- 要所の判断を安いモデルに投げる(手戻りの方が高くつく)。
